# 阶段 08 — 建立结构化 docs/ 目录

## 提示词

```
你是文档架构师。为此项目设计 docs/ 目录结构。

输出：
1. 目录树（含文件名）
2. 每个文件的模板骨架（标题 + 章节占位）

文档类型必须覆盖：
- 架构决策记录（ADR）
- API 参考文档
- 本地开发指南
- 部署运维手册
- 变更日志（CHANGELOG.md）

原则：每个文档只承担一个职责。
```

## 使用说明

- 结构化 docs/ 是"文档园丁" Agent（阶段 22）后续工作的前提
- "每个文档只承担一个职责"防止文档变成信息垃圾场

# 阶段 09 — 编写系统设计与架构文档

## 提示词

```
你是系统架构师。为 [项目名] 编写架构文档 docs/architecture.md。

必须包含：
1. 系统上下文图（用 Mermaid 绘制）
2. 核心模块划分与职责
3. 数据流描述
4. 关键技术决策及其理由（ADR 格式）
5. 已知限制与技术债务
6. 未来演进路径

写作要求：
- 面向新入职工程师
- 避免过度设计描述
- 每个决策说明"为什么不选另一个方案"
```

## 使用说明

- ADR（Architecture Decision Record）格式让历史决策可追溯
- "为什么不选另一个方案"是防止后来者（包括 Agent）重复推翻决策的关键
- Mermaid 图可直接在 GitHub/GitLab 渲染

# 阶段 10 — 划定严格业务层级边界

## 提示词

```
你是领域驱动设计专家。为此项目定义层级边界规则。

项目使用以下分层：[domain / application / infrastructure / interface]

为每一层生成：
1. 允许依赖的层（依赖方向）
2. 禁止依赖的层
3. 允许出现的代码类型（实体、用例、仓储接口等）
4. 违规示例（反面教材，加注释说明为何错误）

同时生成一个自动检测层级违规的 lint 规则或测试用例。
```

## 使用说明

- 自动检测层级违规，让规则有"牙齿"而不只是文档
- 违规示例（反面教材）比正面示例更有约束力
- 依赖方向：interface → application → domain，infrastructure 实现 domain 接口

# 阶段 11 — 编写自定义 Linter 与结构化测试

## 提示词

```
你是质量工程师。为此项目编写：

A. 自定义 Lint 规则（基于 [ESLint / Pylint / custom AST]）：
   - 检测层级依赖违规
   - 检测未处理的 Promise/异常
   - 检测过长函数（超过 50 行警告）

B. 结构化测试模板：
   - 单元测试（AAA 格式：Arrange / Act / Assert）
   - 集成测试骨架
   - Contract 测试示例

每条规则和测试附注释：说明检测的问题和修复方向。
```

## 使用说明

- 自定义 Lint 是团队规范的可执行版本
- AAA 格式（Arrange/Act/Assert）让测试可读性和可维护性一致
- Contract 测试用于跨服务接口稳定性验证
