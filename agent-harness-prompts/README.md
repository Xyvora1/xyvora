## 使用方法

1. 按阶段顺序执行，前序阶段的输出是后续阶段的输入
2. 将所有 `[占位符]` 替换为项目实际值后再使用
3. `AGENTS.md`（阶段 07）生成后，后续所有 Agent 提示词都应引用它

## 核心设计原则

### 1. 占位符模式
所有提示词用 `[技术栈]`、`[task_id]` 等变量标注，同一套模板适配不同项目。

### 2. 输出格式强制声明
每条提示词明确输出格式（YAML / JSON / Markdown / shell 命令），让 Agent 输出可被机械处理。

### 3. 防止 Agent 越界的关键句
- 基础设施阶段："只输出命令，不要解释"
- 执行阶段："如遇歧义，停止并输出问题，不要假设"
- 维护阶段："不要修改任何文件"

### 4. 人工介入触发点
每个关键决策节点都明确输出阻塞信号，等待人工，不让 Agent 自行决断：
- 不同意审查意见 → 输出反驳理由，等待人工裁决
- 合并条件未满足 → 输出阻塞原因，通知人工
- 高风险清理 → 标注 ⚠️，需人工确认

## 阶段分类

### 基础设施（阶段 01-06、11）
建立工程底座：仓库、目录、CI、格式化、包管理、脚手架、测试

### Agent 规范（阶段 07-10）
定义 Agent 行为边界：AGENTS.md、文档结构、架构文档、层级边界

### 自治执行（阶段 12-20）
Agent 自主完成开发循环：工具接入、任务计划、编码、验证、PR、互审、合并

### 持续维护（阶段 21-22）
后台持续扫描：代码垃圾回收、文档过期检测
